# README.md

이 문서는 GEMINI를 사용해 "Petting API" 프로젝트의 코드를 분석하여, 이 코드를 만든 개발자의 개인적인 여정과 성장을 기술 블로그 게시글 형태로 정리한 것입니다. 기능 구현보다는 개발자의 가치관, 고민, 그리고 선택의 과정에 초점을 맞춥니다.

## 1. 코드에서 유추한 개발자의 가치관

*   **견고함과 안정성 추구**: FastAPI, SQLAlchemy, 의존성 주입(`dependency-injector`), 환경 변수 분리(`dotenv`) 등 안정적인 백엔드 시스템 구축에 필요한 기술 스택과 아키텍처 패턴을 적극적으로 활용했습니다. 이는 서비스의 안정적인 운영과 유지보수성을 중요하게 생각하는 가치관을 보여줍니다.
*   **확장성과 유연성 지향**: 도메인별 라우터 분리, 어댑터 패턴 적용, 의존성 주입 컨테이너 활용 등은 향후 기능 추가 및 변경에 유연하게 대응하고 시스템을 확장하기 위한 설계적 고민의 결과입니다.
*   **코드 품질 및 개발 생산성 중시**: `pylint`, `isort`와 같은 코드 품질 도구를 사용하고, `mypy-boto3-stubs`와 같은 타입 힌트 관련 라이브러리를 도입한 것은 높은 코드 품질을 유지하고 개발 생산성을 높이려는 노력을 엿볼 수 있습니다.
*   **실용주의적 접근**: 개발 초기 단계에서 CORS `allow_origins=["*"]`와 같이 빠른 개발을 위한 결정을 내리면서도, 추후 개선의 여지를 남겨두는 실용적인 태도를 보입니다.

## 2. 이 프로젝트에서 겪었을 고민

*   **아키텍처 설계의 복잡성**: 다양한 외부 서비스(AWS, Kakao, Aligo) 연동, 데이터베이스 관리, 인증/인가 처리 등 여러 도메인을 아우르는 백엔드 시스템을 설계하면서, 각 모듈의 책임과 역할을 어떻게 분리하고 통합할지에 대한 깊은 고민이 있었을 것입니다. 특히 `dependency-injector`를 도입한 것은 이러한 복잡성을 관리하기 위한 중요한 결정이었을 것입니다.
*   **환경별 설정 관리**: 개발, 테스트, 배포 환경에 따라 달라지는 API 키, 데이터베이스 URL 등을 어떻게 안전하고 효율적으로 관리할지에 대한 고민이 있었을 것입니다. `dotenv`와 `ExecuteEnviorment` 클래스를 통해 이 문제를 해결하려 했습니다.
*   **개발 속도와 품질의 균형**: 사이드 프로젝트의 특성상 빠른 구현과 기능 검증이 중요하면서도, 장기적인 관점에서 코드 품질과 유지보수성을 놓치지 않으려는 고민이 있었을 것입니다. `pylint` 등의 도구 사용은 이러한 균형을 맞추려는 노력의 일환입니다.
*   **미완성 기능에 대한 결정**: `assessment_router`가 주석 처리된 것을 보면, 특정 기능의 우선순위 조정이나 구현 난이도로 인해 잠시 보류되거나 방향을 재설정하는 과정에서 고민이 있었을 것으로 보입니다.

## 3. 선택의 흔적이 보이는 부분

*   **FastAPI 선택**: 비동기 처리와 높은 성능, 그리고 자동 문서화 기능을 고려하여 FastAPI를 메인 프레임워크로 선택한 흔적이 보입니다.
*   **`dependency-injector` 도입**: 복잡한 의존성 관리를 수동으로 처리하는 대신, 프레임워크를 도입하여 코드의 결합도를 낮추고 테스트 용이성을 높이려는 명확한 선택입니다.
*   **어댑터 패턴 적용**: Kakao, AWS, Aligo 등 외부 서비스와의 연동을 `adapter` 디렉토리 아래에 모듈화한 것은, 외부 서비스의 변경이 내부 로직에 미치는 영향을 최소화하려는 설계적 선택입니다.
*   **`ConfigManager`의 활용**: 애플리케이션 초기화 시 데이터베이스, 어댑터, ORM, 라우터 등을 동적으로 패치하는 방식은 설정의 중앙 집중화와 유연한 관리를 위한 선택입니다.

## 4. 실패 또는 개선의 흔적

*   **`assessment_router` 주석 처리**: 초기 기획 단계에서 고려되었으나, 구현상의 어려움이나 우선순위 조정으로 인해 잠시 보류되거나 방향이 변경된 기능일 수 있습니다. 이는 개발 과정에서 흔히 겪는 시행착오와 개선의 흔적입니다.
*   **CORS `allow_origins=["*"]`**: 개발 편의성을 위해 모든 출처를 허용했지만, 보안 강화를 위해 특정 도메인으로 제한하는 방향으로 개선될 여지가 있습니다. 이는 초기 단계의 빠른 구현과 이후의 점진적 개선을 염두에 둔 흔적으로 볼 수 있습니다.
*   **`requirements.txt`의 주석 처리된 `urllib3`**: 특정 버전 충돌이나 불필요한 의존성 제거 과정에서 발생한 흔적일 수 있습니다.

## 5. 성장 포인트

*   **아키텍처 설계 능력 향상**: 모듈화, 의존성 주입, 어댑터 패턴 등 다양한 설계 패턴을 적용하며 복잡한 시스템을 구조화하는 능력이 크게 성장했을 것입니다.
*   **클린 코드 및 유지보수성 고려**: 코드 품질 도구 사용, 환경 변수 분리, 타입 힌트 활용 등을 통해 단순히 기능 구현을 넘어 클린 코드와 유지보수성을 고려하는 개발 습관을 정립했을 것입니다.
*   **문제 해결 및 의사 결정 능력**: 개발 과정에서 발생하는 다양한 기술적 난관(예: 외부 서비스 연동, 설정 관리)을 해결하고, 개발 속도와 품질 사이에서 균형 잡힌 의사 결정을 내리는 경험을 통해 문제 해결 능력이 향상되었을 것입니다.
*   **다양한 기술 스택 경험**: FastAPI, SQLAlchemy, AWS, Kakao API 등 여러 기술 스택을 통합하여 하나의 서비스를 구축하는 경험을 통해 폭넓은 기술 이해와 활용 능력을 갖추게 되었을 것입니다.

## 6. 개인 스토리 초안 (500자)

"펫팅 API"는 한 개발자의 열정과 성장이 고스란히 담긴 프로젝트입니다. 그는 단순히 기능을 구현하는 것을 넘어, 견고하고 확장 가능한 시스템을 만들고자 했습니다. 처음에는 FastAPI의 매력에 빠져 빠른 개발을 꿈꿨지만, 이내 복잡한 외부 서비스 연동과 데이터베이스 관리라는 현실적인 벽에 부딪혔습니다. 카카오 로그인, AWS S3 연동, 알리고 문자 발송 등 다양한 외부 의존성을 어떻게 깔끔하게 관리할지 고민했고, 그 결과 `dependency-injector`와 어댑터 패턴을 도입하며 아키텍처 설계 능력을 한 단계 끌어올렸습니다.

개발 과정에서 `assessment_router`처럼 잠시 멈춰선 기능도 있었지만, 이는 실패가 아닌 더 나은 방향을 찾기 위한 고민의 흔적이었습니다. 환경 변수를 분리하고, 코드 품질 도구를 적용하며, 그는 단순히 동작하는 코드를 넘어 '잘 만들어진 코드'의 가치를 깨달았습니다. 이 프로젝트는 그에게 기술적 깊이뿐만 아니라, 문제 해결과 의사 결정의 중요성을 가르쳐주었습니다. "펫팅 API"는 단순한 사이드 프로젝트가 아니라, 한 개발자가 끊임없이 배우고 성장하며 자신만의 개발 철학을 정립해나간 여정의 기록입니다.

## 7. 한 줄 캐릭터 정의
